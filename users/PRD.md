# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트 명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- Frontend: HTML, CSS, React, Javascript
- Backend: Node.js, Express
- Database: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음 

## 4. 프런트엔드 PRD - 주문하기 화면

### 4.1 목표
- 사용자가 **메뉴를 선택**하고 **장바구니를 구성**한 뒤 **주문을 요청**할 수 있다.

### 4.2 화면 구성(와이어프레임 기준)
- **상단 내비게이션**
  - 좌측: 서비스 로고/타이틀(예: COZY)
  - 우측: 탭/링크 2개
    - **주문하기**(현재 화면, 활성 상태)
    - **관리자**(관리자 화면으로 이동)
- **메뉴 목록 영역(카드 그리드)**
  - 카드 기본 요소
    - 이미지(썸네일)
    - 메뉴명(예: 아메리카노(ICE), 아메리카노(HOT), 카페라떼)
    - 기본 가격(원 단위)
    - 간단 설명(텍스트)
    - 옵션 체크박스 목록(예: 샷 추가 +500원, 시럽 추가 +0원)
    - **담기** 버튼
- **장바구니 영역(요약 패널)**
  - 제목: 장바구니
  - 담긴 항목 리스트(행 단위)
    - 표시 정보: 메뉴명 + 선택 옵션, 수량(x N), 행 금액
  - 우측 요약
    - **총 금액**
    - **주문하기** 버튼

### 4.3 핵심 사용자 시나리오
- **메뉴 담기**
  - 사용자가 메뉴 카드에서 옵션(체크박스)을 선택 후 **담기**를 누른다.
  - 동일한 메뉴/동일한 옵션 조합이 이미 장바구니에 있으면 **수량만 증가**한다.
  - 다른 옵션 조합이면 **별도 항목으로 추가**한다.
- **총액 확인 후 주문**
  - 장바구니의 각 행 금액과 총 금액을 확인한다.
  - **주문하기**를 누르면 주문 생성 요청을 보낸다.

### 4.4 가격/수량 계산 규칙(UI 표시 포함)
- **옵션 가격 합산**
  - 행 단가 = 기본 가격 + (선택된 옵션들의 추가금 합)
  - 행 금액 = 행 단가 × 수량
- **총 금액**
  - 총 금액 = 모든 행 금액의 합
- **표시 형식**
  - 모든 금액은 `#,###원` 형식으로 표시(예: 12,500원)

### 4.5 UI 동작/상태 정의
- **초기 상태**
  - 메뉴 목록은 로딩 후 카드로 표시
  - 장바구니는 비어있을 수 있음(빈 상태 문구 표시는 선택)
  - 장바구니가 비어있으면 **주문하기 버튼 비활성화**
- **로딩 상태**
  - 메뉴 로딩 중: 스켈레톤/로딩 문구 중 택1
  - 주문 요청 중: 주문하기 버튼 로딩/비활성(중복 클릭 방지)
- **오류 상태(예: 네트워크 실패)**
  - 메뉴 로딩 실패: 재시도 안내(버튼 또는 문구)
  - 주문 실패: 실패 메시지 표시, 장바구니 유지
- **성공 상태**
  - 주문 성공: 성공 메시지 표시 후 장바구니 비우기(또는 주문 번호 표시 후 비우기)

### 4.6 입력 제약 및 검증(프런트 기준)
- **수량**
  - 최소 1, 정수만 허용
  - UI상 수량 조절 컴포넌트는 와이어프레임에 없으므로 1단계에서는 “담기 시 동일 항목 수량 증가”로 충족
- **옵션**
  - 옵션은 복수 선택 가능(체크박스)

### 4.7 접근성/UX 요구사항(최소)
- 버튼/체크박스는 키보드 탭 이동 가능해야 함
- 활성 탭(주문하기/관리자)은 시각적으로 구분(aria-current 등 포함 권장)
- 이미지가 없거나 로딩 실패 시 대체 텍스트 제공

### 4.8 잠재 리스크(누락 시 오류/영향도)
- **동일 항목 판별 기준 불명확**: 메뉴ID만으로 합치면 옵션이 섞여 잘못 합산될 수 있음 → “메뉴ID + 옵션 조합” 기준을 명시(4.3).
- **가격 계산 기준 누락**: 옵션 추가금/0원 옵션 처리 미정 시 총액이 틀릴 수 있음 → 4.4 규칙 고정.
- **중복 주문 요청**: 주문 버튼 연타로 중복 주문 생성 가능 → 주문 요청 중 비활성/로딩(4.5).
- **빈 장바구니 주문**: 서버 오류/불필요 요청 → 버튼 비활성(4.5).

## 5. 프런트엔드 PRD - 관리자 화면

### 5.1 목표
- 관리자가 **주문 현황을 확인**하고 **주문 상태를 처리**할 수 있다.
- 관리자가 **메뉴별 재고를 증감**할 수 있다.

### 5.2 화면 구성(와이어프레임 기준)
- **상단 내비게이션**
  - 좌측: 서비스 로고/타이틀(예: COZY)
  - 우측: 탭/링크 2개
    - **주문하기**(주문하기 화면으로 이동)
    - **관리자**(현재 화면, 활성 상태)
- **관리자 대시보드(요약 카드)**
  - 지표 4개를 한 줄로 표시
    - 총 주문
    - 주문 접수
    - 제조 중
    - 제조 완료
- **재고 현황(메뉴 카드 목록)**
  - 메뉴별 카드 기본 요소
    - 메뉴명(예: 아메리카노(ICE), 아메리카노(HOT), 카페라떼)
    - 현재 재고 수량(예: 10개)
    - 수량 조절 버튼: `+`, `-`
- **주문 현황(리스트)**
  - 주문 행 표시 정보
    - 주문 시간(예: 7월 31일 13:00)
    - 주문 내역 요약(메뉴명 × 수량)
    - 주문 금액
    - 액션 버튼(와이어프레임: **주문 접수**)

### 5.3 핵심 사용자 시나리오
- **대시보드 지표 확인**
  - 관리자는 현재 전체 주문 흐름을 숫자로 빠르게 확인한다.
- **재고 조정**
  - 관리자는 메뉴별 `+/-` 버튼으로 재고를 증감한다.
- **주문 처리**
  - 관리자는 주문 목록에서 주문을 선택해 상태를 진행한다(예: 접수).

### 5.4 주문 상태 및 전이(프런트 표시 기준)
- **상태 정의(최소)**
  - 주문 접수(접수됨)
  - 제조 중
  - 제조 완료
- **대시보드 집계 규칙**
  - 총 주문: 전체 주문 수
  - 주문 접수/제조 중/제조 완료: 각 상태별 주문 수
- **상태 전이(초안, 1단계)**
  - 와이어프레임에 “주문 접수” 버튼만 존재하므로, 1단계에서는 아래 중 택1로 구현 가능
    - (안 A) 주문 상태가 “주문 접수 대기” → “주문 접수”로 변경
    - (안 B) 주문 상태가 “주문 접수” → “제조 중”으로 변경
  - 상태 전이는 PRD 확정 시 **단일 정책으로 고정**해야 하며, 대시보드 집계와 일치해야 한다.

### 5.5 재고 증감 규칙(프런트 기준)
- **증감 단위**
  - `+` 클릭: 재고 +1
  - `-` 클릭: 재고 -1
- **최소값**
  - 재고는 0 미만으로 내려갈 수 없음(0에서 `-`는 비활성 또는 에러 메시지)
- **표시**
  - 재고는 `N개` 형식으로 표시

### 5.6 UI 동작/상태 정의
- **초기 상태**
  - 대시보드/재고/주문 목록을 로딩 후 표시
- **로딩 상태**
  - 화면 진입 로딩: 스켈레톤/로딩 문구 중 택1
  - 재고 증감 요청 중: 해당 메뉴 카드의 `+/-` 비활성(연타 방지)
  - 주문 처리 요청 중: 해당 주문 행의 버튼 비활성(연타 방지)
- **오류 상태(예: 네트워크 실패)**
  - 대시보드/재고/주문 로딩 실패: 재시도 안내
  - 재고 업데이트 실패: 실패 메시지 표시, 화면 값은 서버 기준으로 롤백/재조회
  - 주문 상태 변경 실패: 실패 메시지 표시, 화면 값은 서버 기준으로 롤백/재조회

### 5.7 접근성/UX 요구사항(최소)
- `+/-` 및 주문 처리 버튼은 키보드로 접근 가능해야 함
- 활성 탭(관리자)은 시각적으로 구분(aria-current 등 포함 권장)

### 5.8 잠재 리스크(누락 시 오류/영향도)
- **상태 전이 정책 미확정**: 버튼 라벨/대시보드 집계/주문 리스트 상태가 서로 불일치할 수 있음 → 5.4에서 단일 정책 확정 필요.
- **재고 음수/연타 경합**: `-` 연타/동시 업데이트로 음수 표시 또는 서버와 불일치 가능 → 0 하한, 요청 중 비활성, 실패 시 재조회(5.5~5.6).
- **리스트 갱신 규칙 누락**: 주문 처리 후 목록/대시보드가 즉시 갱신되지 않으면 운영 혼선 → 성공 시 관련 영역 재조회 또는 로컬 업데이트 규칙 명시 필요.
